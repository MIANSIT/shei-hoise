Section 1

What Is This System?

Simple, manual subscription management for the first version of Shei Hoise. No payment gateway. No automation. You are in full control.

Core Idea in One Line: Every new store gets 7 free trial days. After that, YOU (the admin) decide who gets access — by calling the customer, taking payment manually (bKash/bank), and clicking Activate.

Why Keep It Simple for MVP?

🚀
Build in Days, Not Weeks
No payment gateway setup, no webhook complexity, no API keys from SSLCommerz needed right now.
🤝
Personal Customer Handling
For a new product, personally calling each customer builds trust and helps you understand their needs.
🧠
You Stay in Control
If a customer's business is busy, you can give them a few extra days before blocking. No rigid automation.
🔧
Easy to Upgrade Later
The database structure is ready for automated payments. You just add SSLCommerz on top later.

What You Are NOT Building in MVP

✗  No SSLCommerz or payment gateway
✗  No /pricing page with "Pay Now" button
✗  No automatic trial expiry cron job
✗  No webhook or payment callback

All of this comes in Version 2 after you have paying customers.

Step-by-Step Detail

STEP 1 — SIGNUP
Customer creates a store
Goes through /sign-up → /onboarding. Database automatically sets: status = 'trial', is_active = true, trial_ends_at = today + 7 days
↓
STEP 2 — FREE TRIAL (Days 1–7)
Full access to dashboard
Customer uses products, orders, customers — everything. A countdown banner shows them how many days are left in their trial. e.g. "Trial expires in: 5d 12h 30m"
↓
STEP 3 — TRIAL ENDS (Day 7+)
You are notified. Nothing auto-blocks yet.
In the admin panel, the store shows as "Trial Ended" with a yellow warning. You can see exactly when the trial expired. Store still has access during this period.
↓
STEP 4 — YOU CALL THE CUSTOMER
Discuss subscription plan, take payment manually
Customer pays via bKash, bank transfer, or cash. You verify the payment yourself. If they need more time, you can set the grace period or add a note in the admin panel.
↓
STEP 5A — PAYMENT RECEIVED
You click "Activate" in admin panel
One button click. Store becomes status = 'active', is_active = true. Customer gets full access. You note: "Paid ৳599 via bKash — 30 days"
STEP 5B — NO PAYMENT (3+ days)
You click "Block" in admin panel
One button click. Store becomes status = 'expired', is_active = false. Customer is redirected to a contact page when they visit dashboard.

Section 5

Access Control

How the system decides who gets into the dashboard. Enforced on the server — cannot be bypassed.

The Rule Is Simple

Only one condition to check: Is stores.is_active = true?

— true → allow dashboard access
— false → redirect to a "Contact Us to Restore Access" page

Two Layers of Protection

🔐
Layer 1: Next.js Middleware (Root)
File: middleware.ts

Runs on the server before any page loads. Checks is_active from the database on every single request to /dashboard/*.

Even if someone knows JavaScript and tries to bypass the UI — they still get redirected. This is the real security.
🖥️
Layer 2: Dashboard Layout
File: src/app/dashboard/layout.tsx

Already exists. Shows TrialEnded or AccessRestricted screens when storeIsActive = false.

This is a visual fallback. Middleware handles the real blocking.

What the Blocked User Sees

BLOCKED STORE
Customer tries to visit /dashboard
Middleware detects is_active = false → immediately redirects to /contact-us (existing page). Customer sees: "Your store access has been suspended. Contact us to reactivate." — with your phone number and email.
Why redirect to /contact-us and not /pricing? Because in this MVP, there is no self-service payment page. The customer must contact you to pay manually. You can update this link to /pricing later when you add automated payments.
Section 6

Database Changes

Minimal changes to your existing database. No new complex tables for MVP.

Existing tables NOT touched: products, orders, customers, expenses — all untouched.
Only stores table gets 3 new columns. That's it for MVP.

Changes to stores Table

ColumnTypeWhat It StoresStatus
status TEXT (enum) Now: trial, active, expired, grace, cancelled (was: trial, approved, rejected, pending) Updated
is_active BOOLEAN Already exists. Controls dashboard access. true = open, false = blocked. Existing
trial_ends_at TIMESTAMPTZ Exact date/time the 7-day trial expires. Set when store is created. New
subscription_expires_at TIMESTAMPTZ When the paid subscription expires (set by admin when activating). e.g. today + 30 days. New
admin_notes TEXT Free text notes by admin. e.g. "Paid ৳599 bKash on 25 Apr, ref TXN123" New

SQL to Run in Supabase SQL Editor

-- Step 1: Add new columns to stores table ALTER TABLE stores ADD COLUMN IF NOT EXISTS trial_ends_at TIMESTAMPTZ, ADD COLUMN IF NOT EXISTS subscription_expires_at TIMESTAMPTZ, ADD COLUMN IF NOT EXISTS admin_notes TEXT; -- Step 2: Backfill trial_ends_at for existing trial stores UPDATE stores SET trial_ends_at = created_at + INTERVAL '7 days' WHERE status = 'trial' AND trial_ends_at IS NULL; -- Step 3: Add new status values to enum -- (run these one at a time if your status is an enum type) ALTER TYPE store_status ADD VALUE IF NOT EXISTS 'active'; ALTER TYPE store_status ADD VALUE IF NOT EXISTS 'expired'; ALTER TYPE store_status ADD VALUE IF NOT EXISTS 'grace'; ALTER TYPE store_status ADD VALUE IF NOT EXISTS 'cancelled';
If status is TEXT not ENUM: Skip Step 3. Just use the string values directly in your code. Check Supabase Table Editor to confirm the column type.
Section 7

Files to Build

Only 5 things to create or change. That's the entire MVP.

Total work: 2 new files, 1 update to existing file, 1 small enum update, 1 SQL query. A developer can finish this in one day.

New Files to Create

  • SERVER
    middleware.ts   (project root, next to package.json)
    Intercepts every /dashboard/* request. Checks is_active from database. Redirects blocked stores to /contact-us. This is the most important file — it makes the protection real.
  • PAGE
    src/app/dashboard/admin/stores/page.tsx
    Super admin panel. Shows all stores in a table with their status, trial end date, notes. Has Activate / Block / +7 Days / Grace buttons.
  • API
    src/app/api/admin/stores/[id]/route.ts
    PATCH endpoint. Handles the admin button actions: activate, block, extend-trial, grace. Only callable by super_admin users.

Existing Files to Modify

  • EDIT
    src/lib/types/enums.ts
    Add ACTIVE, EXPIRED, GRACE, CANCELLED to StoreStatus enum. Update STORE_STATUS_LABELS with these values.
  • EDIT
    src/app/components/admin/StoreStatus/TrialEnded.tsx
    Change "Upgrade & Continue" button link from /contact-us to /contact-us (same, already correct). Optionally update text to show phone number.
  • EDIT
    src/lib/queries/onboarding/ (store creation query)
    When creating a new store, also set trial_ends_at = now() + 7 days in the INSERT query.

Your Daily Workflow as Admin

How you will actually use this system day to day to manage customers.

Daily Check (2 Minutes)

CHECK
Open /dashboard/admin/stores
Look for stores with status TRIAL that have less than 2 days remaining. These customers need a call soon. Also check any GRACE stores that have been waiting too long.

When a Trial is Expiring

SituationYour ActionButton to Click
Trial ending in 2 days Call customer. Remind them about subscription. No button yet — just call
Trial ended, customer asked for 1 more day Set grace period. Note the conversation. Grace + write note
Trial ended, customer can't pay for 7 more days Extend trial if you want to keep them happy +7 Days + write note
Customer paid via bKash — you verified Activate immediately. Write payment reference in note. Activate + write note
3+ days no response, no payment Block the store. Block
Blocked customer contacts you and pays Activate immediately. Store access restored. Activate
Always write a note. The admin_notes field is your record of every payment and conversation. If a customer claims they paid but you have no note — you can check. Write: "Paid ৳299 bKash 01711-xxxxx ref TXN123 on 25 Apr 2026"
