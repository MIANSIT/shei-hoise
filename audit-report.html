Executive Summary
High-level findings and immediate actions required

The Shei Hoise platform is a functioning multi-tenant e-commerce SaaS built on Next.js 16 and Supabase. The system handles orders, inventory, expenses, and provides a dashboard with financial metrics. However, this audit uncovered 5 critical security vulnerabilities, multiple financial calculation errors, and fundamental structural gaps that make accurate hand cash tracking impossible in its current form.

🔴
Immediate Action Required: The Supabase service role key is exposed in browser JavaScript. Combined with zero Row-Level Security policies, any visitor to the site has full admin access to your entire database including all customer and order data across all stores.

Top 5 Issues Requiring Immediate Action

  • Critical  Service role key exposed as NEXT_PUBLIC — full DB admin access from browser
  • Critical  Zero RLS policies — all users can read/write all data across all stores
  • Critical  Auth is a cookie string check — trivially bypassed via DevTools
  • Critical  Inventory race condition — concurrent orders cause overselling
  • Critical  COGS uses current product prices for historical orders — profit history is retroactively wrong

Financial System Status

What Works
  • Order management and status tracking
  • Manual expense entry and categorization
  • Inventory quantity tracking
  • Period-scoped revenue display
  • Product cost price storage (tp_price)
What Does NOT Work
  • Accurate hand cash calculation
  • Historical COGS (changes when prices edited)
  • Inventory purchase tracking
  • Initial capital / investment recording
  • Lifetime financial view (only 7/30/365 days)

Current System Overview
How each module works — files and functions

Sales Flow

1
Cart → Order Token
Customer cart POSTed to src/app/api/generate-order-token/route.ts. A 7-char Redis token is generated using Math.random() (insecure). Token stores cart+price data in Redis with TTL.
2
Order Confirmation
Customer confirms via src/app/api/get-confirm-order/route.ts. Token is redeemed, order is created via createOrder() in src/lib/queries/orders/orderService.ts.
3
DB Write
orders row created with payment_status='pending'. order_items rows created per product. Inventory reserved: quantity_available -= qty.
4
Admin Marks Paid
Admin changes payment_status to 'paid' via src/lib/queries/orders/updateOrderByNumber.ts. No automatic payment capture system exists.

Purchases

❌
Purchases do not exist as a system. There is no table, no API, no UI, and no data model for recording inventory purchases from suppliers. The only cost data is tp_price — a static field on each product. It records what something costs per unit, not when or how much cash you spent buying stock.

Expenses

Manual expense entry exists. Admin logs expenses via the dashboard into the expenses table with fields: amount, title, expense_date, vendor_name, payment_method, category_id. The expenses and expense_categories tables are missing from schema.sql — they were created directly in Supabase.

Hand Cash Analysis
What data exists, what is missing, and how to calculate
⚠️
Direct database access is not available in this audit. The formulas and SQL queries below reflect what the code actually does. Run these in Supabase Studio → SQL Editor with your real store ID to get actual numbers.

Data Source Map

COGS (Cash Out)order_items × products.tp_price Derived, not recorded Approximate

Hand Cash Formula (Current System)

Hand Cash ≈
  Total Revenue  = SUM(orders.total_amount − shipping_fee) [payment_status='paid']
  − Total COGS  = SUM(order_items.qty × products.tp_price) [paid orders, current prices]
  − Total Expenses  = SUM(expenses.amount) [manually entered]

NOT INCLUDED: Initial Investment · Courier Costs · Unsold Inventory Purchases

SQL to Get Actual Numbers

-- Run in Supabase Studio → SQL Editor
-- Replace 'your-store-id' with your actual store UUID

-- 1. CASH IN: Total revenue from all paid orders (all-time)
SELECT
  COUNT(*) AS paid_orders,
  SUM(total_amount - COALESCE(shipping_fee, 0)) AS total_revenue
FROM orders
WHERE payment_status = 'paid'
  AND store_id = 'your-store-id';

-- 2. CASH OUT (approx): COGS for all sold products
SELECT
  SUM(oi.quantity * COALESCE(pv.tp_price, p.tp_price, 0)) AS total_cogs
FROM order_items oi
JOIN orders o ON o.id = oi.order_id
LEFT JOIN product_variants pv ON pv.id = oi.variant_id
LEFT JOIN products p ON p.id = oi.product_id
WHERE o.payment_status = 'paid'
  AND o.store_id = 'your-store-id';

-- 3. CASH OUT: All expenses
SELECT SUM(amount) AS total_expenses, COUNT(*) AS expense_count
FROM expenses
WHERE store_id = 'your-store-id';

-- hand_cash ≈ total_revenue - total_cogs - total_expenses
-- NOTE: Does not include initial investment or courier payments
ℹ️
Products with NULL tp_price: Any product missing a cost price contributes 0 to COGS, making that product appear 100% profitable. Check how many products have missing cost prices with: SELECT COUNT(*) FROM products WHERE tp_price IS NULL AND store_id = 'your-store-id';

Root Problems
Structural causes — not just surface bugs

Why Hand Cash Cannot Be Accurately Tracked

  • No transaction ledger. The system records outcomes (an order exists, a product has a cost price) but not events (cash left wallet at 3pm Tuesday). Hand cash requires knowing when money moved, in which direction, and how much.
  • COGS is derived, not recorded. tp_price is a property of a product, not a purchase record. When you restock from a supplier, nothing is written to the database.
  • One-way inventory tracking. product_inventory tracks current quantities only. There is no history of when stock went in or out, or what was paid per batch.
  • Expenses are manual opt-in. If you don't log an expense, it doesn't exist. Automatic cash flows like courier fees have no capture mechanism.
  • No initial capital baseline. Hand Cash = Opening Balance + Cash In − Cash Out. The opening balance field does not exist.

Why Dashboard Numbers May Be Wrong

  • COGS uses today's tp_price for all historical orders. Any product price edit rewrites all past profit history.
  • Deleted products contribute 0 COGS — their orders show 100% profit forever.
  • Products with tp_price = NULL show 100% margin — silently inflating gross profit.
  • Dashboard is period-scoped — no lifetime financial view exists.
  • Expenses cap at 10,000 rows. Beyond that, they're silently missing.
  • Shipping margin is invisible — neither collected fee nor courier cost is tracked.

Redesign Plan
Fix without rebuilding — additive changes only
💡
Core Principle: Keep every existing table and flow. Add a financial_transactions table that records every financial event. No existing code is broken — new code writes to this table alongside existing operations.

New Table 1: financial_transactions

CREATE TABLE financial_transactions (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  store_id        uuid NOT NULL REFERENCES stores(id),
  type            varchar(50) NOT NULL,
  -- Types: 'sale' | 'cogs' | 'expense' | 'purchase' | 'investment'
  --        'refund' | 'shipping_in' | 'shipping_out' | 'adjustment'
  direction       varchar(10) NOT NULL CHECK (direction IN ('in', 'out')),
  amount          numeric(10,2) NOT NULL,
  reference_type  varchar(50),   -- 'order' | 'expense' | 'purchase' | 'manual'
  reference_id    uuid,          -- FK to orders.id, expenses.id, etc.
  description     text,
  transaction_date date NOT NULL DEFAULT CURRENT_DATE,
  created_at      timestamptz DEFAULT now(),
  created_by      uuid
);

CREATE INDEX ON financial_transactions(store_id, transaction_date);
CREATE INDEX ON financial_transactions(store_id, type, direction);

New Table 2: inventory_purchases

CREATE TABLE inventory_purchases (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  store_id      uuid NOT NULL REFERENCES stores(id),
  product_id    uuid REFERENCES products(id),
  variant_id    uuid REFERENCES product_variants(id),
  quantity      integer NOT NULL,
  unit_cost     numeric(10,2) NOT NULL,
  total_cost    numeric(10,2) GENERATED ALWAYS AS (quantity * unit_cost) STORED,
  vendor_name   text,
  purchase_date date NOT NULL,
  payment_method varchar(50),
  notes         text,
  created_at    timestamptz DEFAULT now()
);

New Table 3: capital_injections

CREATE TABLE capital_injections (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  store_id       uuid NOT NULL REFERENCES stores(id),
  amount         numeric(10,2) NOT NULL,
  type           varchar(50) CHECK (type IN ('initial_investment', 'owner_contribution', 'loan')),
  injection_date date NOT NULL,
  notes          text,
  created_at     timestamptz DEFAULT now()
);

Schema Fix: Snapshot Cost Price in order_items

-- Add cost_price column to order_items
ALTER TABLE order_items ADD COLUMN IF NOT EXISTS cost_price numeric(10,2) DEFAULT 0;

-- Backfill existing rows (best-effort approximation)
UPDATE order_items oi
SET cost_price = COALESCE(
  (SELECT pv.tp_price FROM product_variants pv WHERE pv.id = oi.variant_id),
  (SELECT p.tp_price FROM products p WHERE p.id = oi.product_id),
  0
);

Corrected Hand Cash Formula (After Fix)

-- Simple, single query after financial_transactions is populated:
Hand Cash =
  SUM(amount WHERE direction='in') -- sales + investments
  - SUM(amount WHERE direction='out') -- cogs + expenses + purchases

WHERE store_id = ? [no period filter = all-time]

-- ⚠️ This uses today's prices, not historical. Accept as one-time approximation.

Step B — Backfill financial_transactions from paid orders

-- Sale transactions (cash in)
INSERT INTO financial_transactions
  (store_id, type, direction, amount, reference_type, reference_id, transaction_date)
SELECT
  o.store_id, 'sale', 'in',
  o.total_amount - COALESCE(o.shipping_fee, 0),
  'order', o.id, o.created_at::date
FROM orders o
WHERE o.payment_status = 'paid';

-- COGS transactions (cash out) using snapshotted cost_price
INSERT INTO financial_transactions
  (store_id, type, direction, amount, reference_type, reference_id, transaction_date)
SELECT
  o.store_id, 'cogs', 'out',
  SUM(oi.quantity * COALESCE(oi.cost_price, 0)),
  'order', o.id, o.created_at::date
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
WHERE o.payment_status = 'paid'
GROUP BY o.id, o.store_id, o.created_at;

Step C — Backfill from expenses

INSERT INTO financial_transactions
  (store_id, type, direction, amount, reference_type, reference_id, transaction_date)
SELECT
  store_id, 'expense', 'out',
  amount, 'expense', id, expense_date
FROM expenses;
